# bursthttp v0.1.5 — Request batching, adaptive retry

**Release date:** 2026-04-24

### Install

```bash
go get github.com/muxover/bursthttp@v0.1.5
```

---

## Summary

Two features aimed at real-world workload patterns. The Batch API lets you fire many requests in parallel with a single call and get results back in order — no manual goroutine management. Adaptive retry replaces the one-size-fits-all exponential backoff: 429 responses now respect the server's `Retry-After` hint, connection errors retry immediately without any sleep, and 5xx responses still back off exponentially as before.

---

## What's included

### Request Batch API

- **`Client.Batch(fn func(*Batch)) []BatchResult`** — registers requests inside a closure, fans them out as concurrent goroutines, and returns results indexed to match the order they were added. No goroutine management needed by the caller.
- **`Client.BatchWithContext(ctx, fn)`** — same as `Batch` but propagates a shared context; cancelling it cancels all in-flight requests in the batch at once.
- Helper methods on `Batch` cover all HTTP verbs: `Get`, `Post`, `Put`, `Patch`, `Delete`, `GetURL`, `PostURL`. Each internally acquires and auto-releases a pooled `Request`.
- `Batch.Do(req)` accepts a caller-owned `*Request` for custom cases; the caller is responsible for releasing it after the batch returns.
- When `EnableScheduler` is on, each batch goroutine routes through the per-host scheduler automatically — stable latency under load applies to batches too.

### Adaptive retry

- **`Retryer.WaitAdaptive`** — replaces the fixed exponential sleep with three strategies chosen by the previous attempt's outcome:
  - **429 + `Retry-After` header** → sleep exactly the server-specified duration (integer seconds or HTTP-date, capped at `RetryMaxDelay`).
  - **Timeout or connection error** → no sleep; retry immediately.
  - **Everything else (5xx, etc.)** → standard exponential backoff with jitter.
- Wired into `DoWithContext` automatically — no config changes or code changes required. All existing `MaxRetries`, `RetryBaseDelay`, `RetryMaxDelay`, and `RetryableStatus` settings continue to apply.
